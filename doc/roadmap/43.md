# Task 43 — Fix categoria obsoleta nella Lista dopo spostamento dal Catalogo

## Problema

Dopo aver spostato un articolo da una categoria a un'altra nel Catalogo, tornando alla Lista l'articolo appariva ancora sotto la vecchia intestazione di categoria (e nel pool dell'autocomplete con la vecchia label) per un tempo variabile (50–300 ms su iOS).

## Causa

`useMoveItemCategory.onSuccess` chiamava `invalidateQueries` sia su `qk.items()` che su `qk.listItems()`.

- `qk.items()` → refetch immediato perché il CatalogoScreen ha sempre `useItems()` montato come subscriber attivo.
- `qk.listItems()` → marcato solo come stale, ma **non refetchato** perché la ListaScreen era smontata (nessun subscriber attivo). Con `staleTime: Infinity` ([`src/lib/queryClient.ts`](../src/lib/queryClient.ts)), i dati stale non scadono mai automaticamente.

Quando l'utente tornava alla Lista, TanStack Query restituiva i dati stale della cache (comportamento stale-while-revalidate) e avviava un refetch in background: l'interfaccia mostrava la vecchia categoria per la durata del refetch.

## Soluzione

Sostituito `invalidateQueries` con `refetchQueries({ type: 'all' })` per `qk.listItems()`.

`refetchQueries` con `type: 'all'` avvia il refetch **indipendentemente dalla presenza di subscriber attivi**: il cache viene popolato con dati freschi mentre l'utente è ancora sul Catalogo, così quando naviga alla Lista trova i dati già aggiornati.

## File modificati

- `src/hooks/useCatalogo.ts` — `useMoveItemCategory.onSuccess`: `invalidateQueries` → `refetchQueries({ type: 'all' })` per `listItems`
- `package.json` — versione → 0.14.3
